# スライドストーリー作成スキル

ユーザーのリクエストを分析し、PPTX 生成の前段階となるスライドストーリー（構成案）を作成するスキルです。

## 重要: 出力方法

**スライド構成はチャットに直接書かず、必ず `set_scenario` ツールを使ってワークスペースパネルに送信すること。**

```
// 正しい方法: set_scenario ツールを呼ぶ
set_scenario({
  title: "プレゼンテーションタイトル",
  slides: [
    { number: 1, title: "タイトルスライド", bullets: ["サブタイトル"], notes: "" },
    { number: 2, title: "アジェンダ", bullets: ["トピック1", "トピック2"], notes: "" },
    ...
  ]
})

// 間違った方法: チャットにマークダウンで書く ← これはやらない
```

ツール呼び出し後、チャットには「スライド構成をワークスペースに作成しました。確認してPPTX生成しますか？」のような短いメッセージだけ返す。

## いつこのスキルを使うか

- ユーザーが「プレゼンを作って」「スライドにまとめて」と依頼した場合
- 複雑なトピックや長い資料をプレゼンに変換する場合
- まずストーリーを確認してから PPTX を生成したい場合

## ストーリーテリング原則（McKinsey式）

### ピラミッド原則
1. **結論先行**: 各スライドのタイトルはそのスライドの結論・主張を述べる（疑問形や曖昧な表現は避ける）
2. **So What?**: 各スライドの `keyMessage` は「だから何？」に答える一文
3. **MECE**: トピックは漏れなくダブりなく分類する

### スライドタイトルの書き方
- ✅ 良い例: 「Azure AI で開発コストを40%削減できる」（主張が明確）
- ❌ 悪い例: 「Azure AI について」（何を伝えたいか不明）
- ✅ 良い例: 「3つの導入パターンから最適な手法を選択」
- ❌ 悪い例: 「導入パターン一覧」

### ストーリーライン構成
```
1. タイトルスライド (layout: title)
   → プレゼン全体の結論を一言で

2. エグゼクティブサマリー or アジェンダ (layout: agenda)
   → 全体の流れと結論の予告

3-N. 本編 — 各セクションは以下の流れ:
   → セクション区切り (layout: section) — セクションテーマ
   → 主張スライド (layout: cards/stats/comparison) — データ・根拠
   → 詳細スライド (layout: bullets/timeline/diagram) — 具体策

N+1. Key Takeaways (layout: summary)
   → 3つの重要ポイントに絞る
```

## 実行手順

### 1. リクエストの分析

ユーザーの依頼から以下を把握する：

- **トピック**: 何についてのプレゼンか
- **対象者**: 誰向けか（経営層、エンジニア、新入社員、一般向け等）
- **目的**: 情報共有、提案、研修、報告等
- **スライド枚数の希望**: 指定があればそれに従う
- **言語**: ユーザーの言語に合わせる

### 2. 情報の収集（Web検索が利用可能な場合）

web_search ツールが利用可能な場合、以下の優先順位で情報を補完する：

1. **公式ドキュメント**: 製品・サービスの公式情報
2. **最新情報**: 新機能・アップデート・トレンド
3. **データ・統計**: 説得力のある数値情報
4. **ベストプラクティス**: 業界標準のアプローチ

### 3. set_scenario ツールで出力

**必ず `set_scenario` ツールを使い、各スライドに以下を含める：**

| フィールド | 必須 | 説明 |
|-----------|------|------|
| `number` | ✅ | スライド番号 |
| `title` | ✅ | 結論・主張を述べるタイトル（McKinsey式） |
| `keyMessage` | ✅ | このスライドで最も伝えたい一文（So What?） |
| `layout` | ✅ | レイアウト種別（下記参照） |
| `bullets` | ✅ | コンテンツ項目 |
| `notes` | - | スピーカーノート |
| `icon` | - | アイコン名 |

### レイアウト種別

| layout | 用途 | 構造 |
|--------|------|------|
| `title` | タイトルスライド | 大きなタイトル + サブタイトル |
| `agenda` | アジェンダ / サマリー | 番号付き項目リスト |
| `section` | セクション区切り | 大きなテキスト + アクセントバー |
| `bullets` | 箇条書き説明 | タイトル + 箇条書き |
| `cards` | 複数項目の並列表示 | 2-4枚のカード |
| `stats` | 数値・統計ハイライト | 大きな数値 + ラベル |
| `comparison` | 比較・Before/After | 2-3列の比較表 |
| `timeline` | 時系列・ステップ | 横並びフロー |
| `diagram` | 概念図・アーキテクチャ | 自由レイアウト |
| `summary` | まとめ・Key Takeaways | 重要ポイント 3つ |

### 利用可能アイコン

`arrow-trending-up`, `brain`, `building`, `calendar`, `chart`, `checkmark-circle`, `cloud`, `code`, `data-trending`, `document`, `globe`, `lightbulb`, `link`, `lock-closed`, `money`, `people-team`, `rocket`, `search`, `settings`, `shield`, `sparkle`, `star`, `target`, `warning`

**各スライドに適切なアイコンを1つ割り当てること。**

### 出力例

```
set_scenario({
  title: "Azure AI エージェント開発による業務革新",
  slides: [
    {
      number: 1,
      title: "Azure AI エージェントで開発生産性を2倍に",
      keyMessage: "AIエージェントの導入で開発チームの生産性が劇的に向上する",
      layout: "title",
      bullets: ["〜 エージェント開発の最前線と導入戦略 〜"],
      icon: "rocket"
    },
    {
      number: 2,
      title: "3つの観点でAIエージェント導入を推進",
      keyMessage: "技術選定・開発プロセス・運用体制の3軸で計画的に導入すべき",
      layout: "agenda",
      bullets: ["1. AIエージェントの技術選定", "2. 開発プロセスへの組込み", "3. 運用体制の構築"],
      icon: "target"
    },
    {
      number: 3,
      title: "AIエージェント市場は2025年に3倍成長",
      keyMessage: "市場の急成長は導入の好機であることを示している",
      layout: "stats",
      bullets: ["市場規模: $5.2B → $15.7B", "導入企業: 前年比180%増", "開発者満足度: 92%"],
      icon: "data-trending",
      notes: "Gartner 2025レポートより"
    }
  ]
})
```

## 言語ルール

- ユーザーの言語に合わせてスライドを作成する
- 専門用語は原語を併記する（例: 「検索拡張生成（RAG）」）
- サービス名・製品名は公式名称をそのまま使用する

## コンテンツ品質チェック

- [ ] タイトルスライドとまとめスライドが含まれている
- [ ] 各スライドのタイトルが明確で具体的
- [ ] 箇条書きは 1-2 文で簡潔
- [ ] 論理的な流れ（導入→本論→結論）になっている
- [ ] 対象者に適した用語レベル
- [ ] ソース情報がある場合はノートに出典を記載
